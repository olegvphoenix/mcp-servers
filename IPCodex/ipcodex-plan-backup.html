Standalone Product

IPCodex

Author: Oleg Vaitsekhovich · LinkedIn

Commercial product for managing API documentation of physical devices. PDF / Swagger / URL to Markdown conversion, firmware versioning, S3 storage, AI Chat, RAG + MCP for AI-assisted development in modern AI IDEs (Cursor, Antigravity, Windsurf).

Why This Matters

Companies integrating physical devices (IP cameras, access control, POS terminals) spend significant time searching, reading, and understanding documentation from dozens of manufacturers. Documentation comes in various formats, scattered across websites and PDF files, unversioned and inaccessible to AI development tools. Structured Markdown documentation is the foundation for AI assistants (Cursor, Antigravity, Windsurf, GitHub Copilot) to effectively help developers write device integration code.

Problems Today

Format Chaos
PDF manuals, Swagger specs, web pages, vendor wikis. Each vendor has its own format, structure, and portal.
No Versioning
Documentation changes with every firmware release. Developers don't know which API version is current for a specific device in the field.
AI Can't Help
LLMs don't know device-specific APIs. Developers manually copy documentation fragments into chat context.
Coding Blind
AI assistants in IDEs (Cursor, Antigravity, Copilot) have no access to device documentation. Integration code is written without context — with errors in endpoints, parameters, and formats.

What We'll Achieve

Phase 1
Unified Documentation Storage
All device API documentation in one place, in unified Markdown format, versioned by firmware. Automatic conversion from PDF (with OCR), Swagger/OpenAPI, and web pages. Source document language doesn't matter — Chinese, Korean, Japanese, or any other.
Phase 1
Full Audit & Control
Every documentation import is a job with history: who launched it, when, what result, how long it took. Diff between versions shows what changed in the API.
Phase 2
AI Writes Integration Code
Developer in AI IDE asks "how to connect Hikvision camera via ISAPI" — and AI writes working code with real endpoints, parameters, and formats from current documentation. Without IPCodex, LLMs hallucinate non-existent APIs.
Phase 2
AI Chat with Documentation
Built-in chat in the web interface: ask a question — get an answer with citations from specific documents. Context selection: all devices, specific vendor, or firmware.
Phase 2
Multilingual + Translation
Documentation can be in any language. Optional translation to English before conversion — via LLM, DeepL, or Google Translate.
Phase 2
3-5x Development Speedup
Instead of hours searching and reading PDFs — instant semantic search across the entire documentation base. LLM generates integration code based on real API specifications.
Elevator Pitch
IPCodex transforms chaotic physical device documentation (PDF, Swagger, web pages) into a structured knowledge base, enabling AI assistants to write working integration code. No direct market analogues exist.

Защитные барьеры (Moat)

Data moat — накопленная база документации ▼
Каждый импортированный документ увеличивает ценность платформы. Чем больше документации в системе, тем точнее AI-ответы, тем полнее покрытие устройств. Конкурент, стартующий с нуля, не сможет быстро накопить такой объём структурированных данных.
Network effect — Marketplace вендоров ▼
Вендоры публикуют документацию → интеграторы приходят за готовой базой → больше интеграторов привлекает больше вендоров. Классический двусторонний network effect, который усиливается с каждым новым участником.
Switching cost — привязка к платформе ▼
Вся документация, версии прошивок, история импортов, AI-модели, настроенные MCP-серверы в IDE — всё привязано к платформе. Миграция на другое решение означает потерю накопленной базы знаний и переобучение команды.
Domain expertise — знание протоколов устройств ▼
Глубокое знание протоколов ONVIF, ISAPI, OSDP, Wiegand, ОФД, MQTT, Modbus. Конвертация учитывает специфику каждого формата: таблицы команд, XML-схемы, бинарные протоколы. Это не generic-конвертер, а domain-specific инструмент.
First mover — 12-18 мес head start ▼
Прямых аналогов на рынке нет. Ни один существующий продукт (ReadMe, GitBook, Mintlify) не ориентирован на физические устройства, не поддерживает OCR-конвертацию PDF и не предлагает версионирование по прошивкам.
Proven core — ядро уже работает ▼
Ядро конвертации (doc2md-mcp) — production-ready, 242 теста, поддерживает PDF с OCR, Swagger/OpenAPI, веб-страницы. Это не прототип — это работающий инструмент, на базе которого строится платформа.

RAG Pipeline

Step 1
Chunking
MD разбивается по заголовкам (H1/H2/H3). Большие секции — по абзацам с overlap 100 токенов. Сохраняется heading_path.
Step 2
Embedding
Векторизация чанков через OpenAI text-embedding-3-small (1536 dims) или локальную модель all-MiniLM-L6-v2 (384 dims). Размерность настраивается при деплое.
Step 3
pgvector
Хранение в PostgreSQL с расширением pgvector. HNSW индекс для cosine similarity search (работает при любом объёме данных, не требует минимума строк как IVFFlat).
